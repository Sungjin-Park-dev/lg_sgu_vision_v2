# 2026-06-24

UR20에 붙인 **새 custom 카메라**(USD에서만 새로 맞춘 상태)에 맞춰, NVIDIA cuRobo
[`build_robot_model`](https://nvlabs.github.io/curobo/latest/getting-started/build_robot_model.html)
가이드대로 **collision sphere(MorphIt 자동 피팅) + self-collision matrix가 포함된 robot config(yml/xrdf)를
재생성**. 기존 hand-tuned `ur20_with_camera.yml`은 교체. viser로 피팅 결과까지 시각 확인.

## 배경 — 왜 URDF가 필요한가

- cuRobo는 **URDF**로 기구학/충돌 모델을 빌드한다(USD 아님). collision sphere는 URDF link frame에 정의되고
  yml에 link 이름으로 저장된다. → 카메라 충돌을 넣으려면 **URDF에 카메라 mesh link**가 있어야 한다.
  (`plan_trajectory.py:_resolve_robot_config`가 yml의 `urdf_path`만 쓰고 `usd_*`는 버림을 코드로 확인.)
- 사용자는 카메라를 **USD에서만** 새로 맞춘 상태였고, URDF는 구 구조였음. → URDF를 USD에 맞춰 재정의 필요.

## 한 일

### 1. USD 카메라 body mesh → OBJ export

**문제**: 디스크의 `data/camera/camera_asm_asm.obj`는 mm 단위 + 축 순열 + 오프셋이라 URDF에 그대로 못 씀.
반면 `ur20_with_camera.usd`의 camera body Mesh는 이미 **meter 단위로 flange 프레임에 baked**
(local bounds `[-0.001,-0.095,-0.056]~[0.346,0.05,0.056]`).

**해결**: pxr로 USD Mesh(67007 pts)를 추출·triangulate해서 flange 프레임 좌표 그대로
`ur20_description/camera/camera_body.obj`로 export. scale=1, identity origin으로 URDF에 붙이면 USD와 1:1.

### 2. 새 cuRobo URDF 생성 — `ur20_with_camera_curobo.urdf`

기존 `ur20_with_camera.urdf`를 ElementTree로 가공:

- **`world` link + `base_joint`(fixed, origin identity) 제거** → root=`base_link`.
  (RobotBuilder는 URDF root를 base_link로 자동 채택하며 override 불가.)
- 구 카메라 체인 제거(`camera_cable_frame/frame_1/frame_2/camera_link/optical` + joint들).
- 새 카메라 블록: `flange ─(identity)→ camera_link`(visual+collision = `camera_body.obj`)
  `─→ camera_optical_frame`(xyz `0.346 0 0`, rpy `π/2 0 π/2` = USD optical 회전).
- arm mesh 경로(`package://ur_description/meshes/ur20/...`)는 유지.

**검증**: yourdfpy FK로 `camera_optical_frame` rel flange = USD와 **0.000 mm / 0.000°** 일치.

### 3. asset 디렉토리 스테이징

**문제**: 호스트엔 `/curobo` 컨테이너가 없어 URDF의 `package://ur_description/meshes/...`가 resolve 안 됨
(venv curobo엔 ur10e만, ur20/camera 없음).

**해결**: scratchpad에 `<stage>/ur_description/meshes/` 트리 생성 —
`ur20`→repo `ur20_description/ur20` symlink, `camera/camera_body.obj`. 빌더에 `--asset-path <stage>`.

### 4. build_robot_model 실행

```bash
.venv/bin/python -m curobo.examples.getting_started.build_robot_model \
  --urdf ur20_description/ur20_with_camera_curobo.urdf \
  --asset-path <stage> \
  --tool-frames camera_optical_frame \
  --output ur20_description/ur20_with_camera.gen.yml \
  --export-xrdf --compute-metrics --seed 42
```

결과: **180 spheres / 8 link** (base_link_inertia 59, shoulder 30, upper_arm 33, forearm 11,
wrist_1/2/3 4/3/11, **camera_link 29**). self-collision ignore matrix 14 entries. CUDA 사용(헤드리스 OK).
빌더는 visual mesh(`.dae`/`.obj`)에 피팅 — 호스트 viser 0.2.11에서 `.dae` 로딩 정상.

### 5. 출력 post-edit (downstream 호환, 필수)

빌더 출력은 `kinematics:`가 top-level이고 cspace가 기본값이라 그대로는 못 씀:

- **`kinematics:` 블록을 `robot_cfg:` 아래로 래핑** — 안 하면 `_resolve_robot_config`가 `KeyError: 'robot_cfg'`.
- cspace 튜닝값 복원: `default_joint_position: [0.0,-2.2,1.9,-1.383,-1.57,0.0]`(빌더는 midpoint 0),
  `max_acceleration: 12.0`(빌더 10), `position_limit_clip: 0.1`(빌더 0). `.xrdf`도 동일 패치.

### 6. 검증 & 승격

- `_resolve_robot_config(DEFAULT_ROBOT_CONFIG)` 로드 OK(robot_cfg KeyError 없음), 6DOF joint 순서 정상,
  tool=`camera_optical_frame`, sphere 180(camera 29).
- cuRobo FK ↔ yourdfpy(동일 URDF) **0.0001 mm** 일치 — 기구학 배선 정상.
- `*.gen.yml`/`.gen.xrdf` → canonical `ur20_with_camera.yml`/`.xrdf`로 승격
  (yml의 `urdf_path`는 `ur20_with_camera_curobo.urdf` 가리킴).

### 7. viser 시각화

**문제**: 가이드의 `--visualize`는 cuRobo `ViserVisualizer`를 쓰는데 설치된 **viser 0.2.11**과 API 불일치
(`ViserUrdf(load_collision_meshes=...)` 인자 없음, `add_batched_meshes_simple` 없음). viser를 올리면
viewpoint_studio가 위험.

**해결**: 0.2.11 API로 **커스텀 뷰어**(`scratchpad/viz_spheres.py`) 작성 — yourdfpy URDF + `ViserUrdf`로
로봇 메시, cuRobo `kinematics.get_robot_as_spheres`로 retract 자세의 콜리전 스피어 180개를 `add_icosphere`(초록)로
표시. `http://localhost:8080`에서 카메라 포함 피팅 확인 완료.

## 주의 / 결정

- **공유 `ur20_with_camera.urdf`는 안 건드림.** `scripts/isaac/usd/check_camera_alignment.py`와
  Isaac URDF import 문서(`docs/setup_isaac_sim.md`)가 구 구조(world root + camera 프레임)를 기대하므로,
  cuRobo용 URDF는 `_curobo.urdf`로 분리하고 yml이 그걸 가리키게 함.
- asset 스테이징 디렉토리는 비영구(scratchpad). 재빌드 시 `ur_description/meshes/{ur20,camera}` 트리를 다시 만들면 됨.
- 재생성 절차/주의는 메모리에도 기록(`curobo-config-regen-build-robot-model`).

## Viewpoint Studio 업데이트

`scripts/apps/viewpoint_studio.py`를 surface 기반 viewpoint 생성/검증용 UI로 정리하고,
클러스터 내부 방문 순서를 자연스러운 검사 모션에 맞게 lawnmower 패턴으로 바꿨다.

### 표면 샘플링

- `viewpoint_studio.py`에서는 sampling을 **surface 전용**으로 고정했다.
  grid 생성 코드는 `scripts/core/generate_viewpoints.py`와 CLI에만 남겨둔다.
- surface 샘플링은 `trimesh.sample_surface_even()` 대신
  **area-weighted 후보점 oversampling + greedy Farthest Point Sampling(FPS)** 로 변경했다.
  목표 개수는 `area / spacing^2`로 계산한다.
- surface spacing을 직접 mm로 조절하지 않고, 카메라 FOV와 overlap에서 계산한다:
  `surface_spacing = min(FOV_width, FOV_height) * (1 - overlap)`.
- `scripts/common/config.py`의 기본 overlap은 `CAMERA_OVERLAP_RATIO = 0.5`로 조정했다.

### Studio UI

- `FOV overlap (%)` 슬라이더를 추가하고, 현재 FOV/overlap/surface spacing을 상태 텍스트로 표시한다.
- `Layers`에 **Surface points** 레이어를 추가해 raw surface `positions`와
  working-distance가 반영된 `camera_positions` 마커를 같이 비교할 수 있게 했다.
- CoACD 결과는 `(object, surface_spacing, coacd_threshold)` 기준으로 캐시한다.
- `Sub-cluster` UI는 `agglomerative`와 `dbscan`만 노출한다.
  CoACD-only는 별도 처리가 필요해 studio UI에서는 제거했고, core/CLI 경로는 유지했다.
- sub-cluster 선택에 따라 관련 옵션만 표시한다.
  `agglomerative`는 `max span (mm)`, `dbscan`은 `eps (mm)`만 노출한다.
- DBSCAN은 studio에서 `eps`만 조절하고, 내부값은 `min_samples=2`, `normal_weight=0.0`으로 고정했다.

### 클러스터링/방문 순서

- CoACD-only baseline을 확인한 뒤, 곡률 때문에 CoACD part 하나만으로는 클러스터가 너무 커지는
  경우가 있어 sub-clustering을 유지하기로 했다.
- agglomerative는 complete linkage + `max_span_mm` 기준으로, 최종 클러스터의 카메라 위치 지름이
  max span을 넘지 않도록 한다.
- 클러스터 내부 ordering에 `lawnmower` 모드를 추가했다.
  각 최종 sub-cluster마다 평균 normal tangent plane을 만들고, 표면점 projection의 2D PCA로
  scan/row 축을 정한 뒤 `row -> reverse row -> row` 형태의 serpentine 경로를 만든다.
- lawnmower row/scan 축은 surface `positions` 기준으로 만들고, 두 시작 방향 중 실제
  `camera_positions` 경로 길이가 짧은 쪽을 선택한다.
- 기존 `graph` ordering은 실제 동작에 맞춰 camera-space nearest-neighbor + 2-opt로 정리했다.
