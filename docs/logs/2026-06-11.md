# 2026-06-11

viser 스튜디오에서 곡면/입체 객체 viewpoint가 "원하는 영역 밖"에 생기는 문제 두 건을 조사.
`sample`은 **material 필터 미적용**이 원인이라 해결, `square_structure`는 **얇은 벽 속 빈 박스**의
안쪽 벽 문제라 보류. 더불어 `sample`의 초록/회색 라벨 혼동을 데이터 차원에서 통일.

## 한 일

### sample — material 필터 적용 (스튜디오가 전체 메시로 생성하던 문제)

**문제**: 스튜디오 `_generate_worker`가 `load_meshes(obj, None)`로 **전체 메시**를 써서, `sample`의
검사대상 외 영역(초록/회색 둘 다)에 viewpoint가 생김. CLI엔 `--material-rgb`가 있지만 스튜디오는 미노출.

**해결**: 모듈 상수 `OBJECT_TARGET_MATERIAL`(`viewpoint_app.py`) 추가 — 오브젝트별 타깃 재질 RGB 맵.
`load_meshes(obj, OBJECT_TARGET_MATERIAL.get(obj))`로 등록 오브젝트는 그 재질 면만 샘플링(미등록은 전체).
표시 메시는 전체(맥락용), viewpoint만 대상 영역에. 검증: 생성 viewpoint 전부 대상 메시 위(거리 0.0mm).

### sample — 초록/회색 컨벤션 통일 (혼동 제거)

**문제**: `sample`은 2재질(초록 0,255,0 / 회색 170,163,158). 초록이 시각적으로 검사영역처럼 보이는데
프로젝트 컨벤션·CLI 예시 12곳은 **회색=검사대상**으로 써서 늘 헷갈림. 실제 검사대상은 **큰 평면(500면)**.

**해결**: 검사대상을 **초록으로 통일**(green=대상, 시각=의미 일치).
- `data/sample/mesh/source.obj`의 **usemtl 2블록 스왑**(line 1854·2355) → 대상 평면 500면이 초록 재질을,
  비대상(경사·가장자리 524면)이 회색을 사용. MTL의 Kd·이름은 그대로(스왑이라 색↔이름 일치 유지).
- `target.h5`/`target.ply`는 **동일 500면**이라 지오메트리 불변 → 재생성 불요.
- `OBJECT_TARGET_MATERIAL["sample"]="0,255,0"`, 활성 `--material-rgb` 예시 전부 `0,255,0`으로 갱신
  (CLI/docs/prev; archive는 이력이라 유지). 검증: `0,255,0` 질의가 500면 평면을 정확히 선택.

### square_structure — 안쪽 벽 문제 (보류)

표면 샘플링이 **얇은 벽 속 빈 박스의 안쪽 벽(면의 63%)**까지 찍어 viewpoint가 안쪽에 생김.
파이프라인 메시는 watertight(닫힘은 원인 아님). voxel 외피 추출은 얇은 벽(~1.2mm)에서
OOM·누수로 실패, 외부 가시성 ray 필터는 오염/손실 트레이드오프로 미완. → **보류**.
상세·재현·재개 후보: [archive/square-structure-inner-walls.md](../archive/square-structure-inner-walls.md).

### 기타

- 스튜디오 `max span (mm)` 기본값 **80 → 100mm**.

## 결정/변경

- **material 필터 = 오브젝트별 하드코딩**(`OBJECT_TARGET_MATERIAL`). 표시는 전체 메시, 생성은 대상만.
  추후 다물체로 늘면 드롭다운 검토(`parse_mtl_file`/`kd_to_rgb` 기존 활용).
- **컨벤션: 초록(0,255,0) = 검사대상.** 데이터(usemtl) 차원에서 통일해 시각=의미 정합. 회색=비대상.
- **square_structure는 데이터 한계(얇은 벽 속 빈 박스).** 코드 우회보다 업스트림 메시 정리가 정공법일 수
  있어 보류. 단일 외피 객체(곡면/원통/`sample`)엔 무관한 문제.
- 커밋 `dd85457` (sample 통일 + 스튜디오 필터 + max span). square 미해결.

## TODO/후속

  [archive/square-structure-inner-walls.md](../archive/square-structure-inner-walls.md) 참고.
- 스튜디오 코드 변경 후 **재시작 필요**(`mesh_cache`가 옛 메시 보유).
- (이월) surface+graph+agglomerative 결과를 `plan_trajectory`로 궤적 검증.

# 2026-06-11 (오후) — Isaac Sim: 물체 선택·기즈모 이동·궤적 생성/테스트

viser는 viewpoint 생성 전용이라, Isaac Sim에서 **물체를 고르고 → 뷰포트 기즈모로 옮기고 →
그 자리에서 궤적 생성·preview**까지 되도록 기존 `pipeline_ui.py`(Generate/Preview/Publish)를 확장.
구현은 빠르게 끝났지만, **USD 좌표/단위 불일치 버그 디버깅**에 대부분의 시간이 들어감(아래).

## 한 일 — 기능

- **`plan_trajectory.py`**: `--object-position X Y Z` / `--object-quat W X Y Z`(robot frame) 추가.
  argparse 직후 `config.TARGET_OBJECT`를 in-place override → `build_camera_poses`·`build_collision_world`가
  옮긴 pose로 계획. 미지정 시 기존 동작(하위호환).
- **`pipeline_ui.py`**: Generate 패널에 **Object 드롭다운 + Load Object**(prim 스왑), **Log Pose** 버튼,
  `_read_object_world_pose()`(prim 월드 transform → robot frame, `z-=MOUNT_HEIGHT`). Generate 시 물체의
  **현재 pose를 읽어** override 인자로 전달, prim 없으면 중단(stale pose 방지).
- **`joint_control.load_target_object`**: 재호출 가능(delete+create)하게. + pose를 `create_prim`의
  world-pose setter 대신 **USD 로컬 transform(`AddTransformOp`)으로 직접 author**(아래 버그 참고).
- **`scripts/isaac/usd/build_object_usd.py`**(신규): `omni.kit.asset_converter`로 `source.obj→source.usd`
  오프라인 1회 변환(`use_meter_as_world_unit=True` + 변환 후 `upAxis=Z` 패치).
- **`scripts/prep/reorient_mesh.py`**(신규): 보정 회전을 `source.obj`에 베이크(+`.bak` 백업, normalize 규칙 유지).
- **`scripts/isaac/usd/inspect_usd.py`**: `upAxis`/`metersPerUnit`/prim별 `xformOp` 덤프 추가(디버깅에 결정적).

## 디버깅 — USD 좌표/단위 함정 (핵심 기록)

1. **단위(보임 안됨/스케일)**: asset_converter 기본 출력이 cm(`metersPerUnit=0.01`)인데 스테이지는 1.0
   → "Mismatched units". 해결: `ctx.use_meter_as_world_unit=True`.
2. **회전 = config가 아니라 메시에**: 기존 `TARGET_OBJECT["rotation"]=[0.7071,0,0,0.7071]`은 **순수 z-yaw**.
   viser는 로컬·Isaac은 월드 프레임이라 둘이 항상 이 yaw만큼 달라 보임. → **config를 identity로** 바꾸고
   방향은 전부 메시/USD에. 이러면 "그냥 올리면 똑바로 + viser==Isaac". (z회전은 bottom 필터 불변이라 sample 안전.)
3. **boot는 맞는데 Load Object만 90° 돌아감**(가장 오래 걸림): 원인 = `source.usd`가 **upAxis=Y로 라벨**됐는데
   지오메트리는 Z-up. 재생 중(버튼) 참조 시 Omniverse가 `xformOp:rotateX:unitsResolve=90`을 **비동기 주입**(boot
   땐 안 붙음). → **USD upAxis를 Z로** 패치하면 보정 op 자체가 안 생김. `build_object_usd`가 이제 자동 처리,
   기존 4개 USD도 패치 완료. (진단: UI에 임시 Diag 버튼 + `_read_object_world_pose`로 USD vs PHYSICS quat 비교 →
   `rotateX:unitsResolve=90` 포착. 정리 시 제거함.)
4. **sample 안 보임**: `data/sample/mesh/source.usd`(옛 hand-made)가 정점은 미터인데 `metersPerUnit=0.01`로
   라벨돼 100배 축소(~2mm). → root 소유라 `chown` 후 `mpu=1.0`+`upAxis=Z` 패치(라벨만).

## 결정/변경

- **방향 규칙: `config.TARGET_OBJECT["rotation"]=identity`**, 물체 방향은 **메시(`reorient_mesh`)에 베이크**.
  → viser(로컬)·Isaac(월드)·계획이 모두 동일. `load_workcell.py`의 독립 하드코딩도 identity로 정렬.
- **USD 규칙: `metersPerUnit=1.0` + `upAxis=Z`**(스테이지와 일치). 안 맞으면 재생 중 `unitsResolve` 보정 op가
  끼어 회전/스케일이 틀어짐. `build_object_usd`가 보장.
- **prim transform은 USD 로컬 op로 author**(physics backend의 world-pose setter 회피).
- 기존 `pipeline_ui` 확장(신규 앱 아님). 디버그용 Diag 버튼·진단 print는 원인 규명 후 제거.

## 한계 / 후속

- **support 받침대 고정**(`config.WALLS`): 물체를 멀리 옮기면 원위치 phantom 장애물 → 테이블 위 소폭 이동에서 사용.
- **viewpoints는 평행이동 불변, 회전 비불변**(생성 시 회전 기준 bottom 필터) → 큰 회전 후엔 viewpoint 재생성.
- 비-sample 물체는 `source.usd`가 없으면 `build_object_usd.py`로 1회 변환 필요.
- (후속) 옮긴 물체로 Generate→Preview 실로봇 검증, sample도 정규 변환 파이프라인으로 일원화 검토.

# 2026-06-11 (오후 2) — Pipeline UI: sim / real 모드 토글

`pipeline_ui.py`가 부팅 시 항상 ROS 브리지+액션그래프를 켜서, 실로봇 없이 순수 시뮬레이션만
돌리기 불편했음. **sim/real 모드를 구분하고 Isaac UI에서 런타임 토글**하도록 확장.

## 한 일

- **`joint_control.py`**: `parse_args`에 `--mode {sim,real}`(기본 `sim`). `start_sim`에
  `enable_ros_bridge=True` 파라미터(브리지 enable을 조건부로 — eager에선 항상 True). `build_action_graph`에
  **idempotency 가드**(기존 `/ActionGraph` prim 있으면 `delete_prim` 후 재생성 → 재호출 안전).
- **`pipeline_ui.py`**: 상단 `Mode` 패널(ComboBox `sim`/`real` + 색 라벨). 단일 진입점
  **`apply_mode(mode)`** — `ActionGraphSwitch.set_active(mode=="real")`로 그래프 tick 토글 +
  Publish 버튼 enable 동기화 + 로그. `_on_publish` **최상단 sim 거부 가드**(silent no-op 금지).
  `main()`은 `play()` 직후 `apply_mode(args.mode)` 호출 → 기본 sim이면 frame 0부터 tick OFF.
- **`docs/setup_isaac_sim.md`**: pipeline_ui 절에 "모드 (sim/real)" 추가.

## 결정/설계

- **eager 채택**(사용자 확정): 부팅 시 브리지+그래프 **항상 생성**, sim은 tick만 끔. 토글이
  노드 생성/삭제 없이 enable 플래그만 뒤집어 **즉시·견고**. lazy(부팅 시 ROS 완전 부재 + 첫 real
  전환 때 런타임 빌드)는 **play() 이후 ROS2+render-product 노드 생성이 Isaac 6.0에서 불안정**이라 기각.
- **핵심 통찰**: Generate(cuRobo subprocess)·Preview(ghost FK)는 이미 ROS 무관 → 두 모드 동일.
  ROS 의존은 액션그래프 미러링 + Publish 둘뿐 → 둘만 게이트.
- sim 거부는 **명시 로그 + 버튼 비활성**(이중 방어). `set_active`가 `noop`로 풀리면 경고 로그(가시화).

## 한계 / 후속

- **eager 잔여**: sim에서도 브리지 확장 로드 + inert DDS participant 존재(연결 불필요). sim 작업은
  ROS 미source 셸 권장(FastDDS 충돌 회피).
- (후속) real 모드 런타임 전환 후 viewport `/joint_states` 미러링 + Publish 실로봇 검증.
