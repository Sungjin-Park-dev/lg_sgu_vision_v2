# 2026-06-11

viser 스튜디오에서 곡면/입체 객체 viewpoint가 "원하는 영역 밖"에 생기는 문제 두 건을 조사.
`sample`은 **material 필터 미적용**이 원인이라 해결, `square_structure`는 **얇은 벽 속 빈 박스**의
안쪽 벽 문제라 보류. 더불어 `sample`의 초록/회색 라벨 혼동을 데이터 차원에서 통일.

## 한 일

### sample — material 필터 적용 (스튜디오가 전체 메시로 생성하던 문제)

**문제**: 스튜디오 `_generate_worker`가 `load_meshes(obj, None)`로 **전체 메시**를 써서, `sample`의
검사대상 외 영역(초록/회색 둘 다)에 viewpoint가 생김. CLI엔 `--material-rgb`가 있지만 스튜디오는 미노출.

**해결**: 모듈 상수 `OBJECT_TARGET_MATERIAL`(`viewpoint_app.py`) 추가 — 오브젝트별 타깃 재질 RGB 맵.
`load_meshes(obj, OBJECT_TARGET_MATERIAL.get(obj))`로 등록 오브젝트는 그 재질 면만 샘플링(미등록은 전체).
표시 메시는 전체(맥락용), viewpoint만 대상 영역에. 검증: 생성 viewpoint 전부 대상 메시 위(거리 0.0mm).

### sample — 초록/회색 컨벤션 통일 (혼동 제거)

**문제**: `sample`은 2재질(초록 0,255,0 / 회색 170,163,158). 초록이 시각적으로 검사영역처럼 보이는데
프로젝트 컨벤션·CLI 예시 12곳은 **회색=검사대상**으로 써서 늘 헷갈림. 실제 검사대상은 **큰 평면(500면)**.

**해결**: 검사대상을 **초록으로 통일**(green=대상, 시각=의미 일치).
- `data/sample/mesh/source.obj`의 **usemtl 2블록 스왑**(line 1854·2355) → 대상 평면 500면이 초록 재질을,
  비대상(경사·가장자리 524면)이 회색을 사용. MTL의 Kd·이름은 그대로(스왑이라 색↔이름 일치 유지).
- `target.h5`/`target.ply`는 **동일 500면**이라 지오메트리 불변 → 재생성 불요.
- `OBJECT_TARGET_MATERIAL["sample"]="0,255,0"`, 활성 `--material-rgb` 예시 전부 `0,255,0`으로 갱신
  (CLI/docs/prev; archive는 이력이라 유지). 검증: `0,255,0` 질의가 500면 평면을 정확히 선택.

### square_structure — 안쪽 벽 문제 (보류)

표면 샘플링이 **얇은 벽 속 빈 박스의 안쪽 벽(면의 63%)**까지 찍어 viewpoint가 안쪽에 생김.
파이프라인 메시는 watertight(닫힘은 원인 아님). voxel 외피 추출은 얇은 벽(~1.2mm)에서
OOM·누수로 실패, 외부 가시성 ray 필터는 오염/손실 트레이드오프로 미완. → **보류**.
상세·재현·재개 후보: [archive/square-structure-inner-walls.md](../archive/square-structure-inner-walls.md).

### 기타

- 스튜디오 `max span (mm)` 기본값 **80 → 100mm**.

## 결정/변경

- **material 필터 = 오브젝트별 하드코딩**(`OBJECT_TARGET_MATERIAL`). 표시는 전체 메시, 생성은 대상만.
  추후 다물체로 늘면 드롭다운 검토(`parse_mtl_file`/`kd_to_rgb` 기존 활용).
- **컨벤션: 초록(0,255,0) = 검사대상.** 데이터(usemtl) 차원에서 통일해 시각=의미 정합. 회색=비대상.
- **square_structure는 데이터 한계(얇은 벽 속 빈 박스).** 코드 우회보다 업스트림 메시 정리가 정공법일 수
  있어 보류. 단일 외피 객체(곡면/원통/`sample`)엔 무관한 문제.
- 커밋 `dd85457` (sample 통일 + 스튜디오 필터 + max span). square 미해결.
