### `select_ik_dp.py` + `publish_trajectory.py` — continuous scan time planning 으로 정리

초기 구현은 viewpoint 주변 `speed_scale` 을 CSV 에 저장하고, `publish_trajectory.py` 가 이를 해석해 시간을 늘리는 구조였음. 그러나 책임이 두 파일에 나뉘어 혼란이 생기고, `scan-window-mm` 이 viewpoint 간격보다 커지면 거의 모든 waypoint 가 감속 구간으로 들어가 전체가 지나치게 느려지는 문제가 있었음.

**최종 결정: continuous scan**
- viewpoint 주변 개별 감속(`scan-window-mm`, `scan-speed-scale`, `speed_scale` CSV 컬럼) 제거.
- `select_ik_dp.py` 가 최종 CSV 의 `time` 컬럼까지 확정.
- `publish_trajectory.py` 는 CSV 의 `time` 을 그대로 `FollowJointTrajectory.time_from_start` 로 보냄.

**`select_ik_dp.py` 변경**
- CLI 단순화:
  - 유지: `--object`, `--num-viewpoints`, `--viewpoints`, `--spacing`, `--output-suffix`
  - 제거: `--robot`, `--num-seeds`, `--ik-batch-size`, `--dbscan-eps`, `--reconfig-threshold`,
    `--resample-mode`, `--scan-window-mm`, `--scan-speed-scale`, `--ee-speed-mm-s`,
    `--ee-angular-speed-deg-s`, `--max-joint-vel`, `--min-segment-dt`
- 제거한 고급 값은 파일 상단 `Pipeline defaults` 로 이동:
  - `ROBOT_CONFIG`, `NUM_IK_SEEDS`, `IK_BATCH_SIZE`, `DBSCAN_EPS_RAD`, `RECONFIG_THRESHOLD_DEG`
  - `RESAMPLE_MODE = "ee"`, `DEFAULT_SPACING_M = 0.01`
  - `EE_SPEED_MM_S = 80.0`, `EE_ANGULAR_SPEED_DEG_S = 30.0`, `MAX_JOINT_VEL_RAD_S = 0.5`, `MIN_SEGMENT_DT_S = 0.05`
- 시간 계산은 segment 별로 다음 값의 최대를 사용:
  - EE 이동거리 / EE 선속도 제한
  - EE 회전각 / EE 각속도 제한
  - joint 최대 변화량 / joint 속도 제한
  - 최소 segment 시간
- 방향 전환 시 흔들림 완화를 위해 corner slowdown 추가:
  - EE polyline corner angle 과 joint-space corner angle 중 큰 값을 사용
  - threshold 초과 waypoint 주변 segment 의 time 을 최대 `CORNER_MAX_SLOWDOWN` 배까지 증가
  - 기본값: `CORNER_ANGLE_THRESHOLD_DEG = 30.0`, `CORNER_MAX_SLOWDOWN = 2.5`
- CSV 파일명에 time planning 조건 포함:
  - 예: `trajectory_dp_ee_s0010_eev50mms_av20dps_jv0p30_corner30d_x2p5.csv`

**`publish_trajectory.py` 변경**
- CLI 를 `--csv` 하나로 축소.
- `--object`, `--num-viewpoints`, `--max-joint-vel` 제거.
- `common.config` 의존 제거.
- CSV 검증 강화:
  - `time` 컬럼 필수
  - joint row 수와 time row 수 일치
  - trajectory row 1개 이상
  - time strictly increasing
- 현재 로봇 위치에서 CSV 첫 waypoint 까지는 내부 상수로 간단히 연결:
  - `MAX_STEP_RAD = 0.1`
  - `APPROACH_MAX_JOINT_VEL_RAD_S = 0.5`
  - `MIN_APPROACH_TIME_S = 0.5`
- CSV 내부 구간은 time 을 재계산하지 않고 보존.

**주의**
- 기존 `time = i * 1.0` dummy CSV 를 새 publisher 로 실행하면 매우 느리게 재생됨.
- 새 `select_ik_dp.py` 로 재생성한 CSV 를 사용하는 전제.

**실행 확인 (sample/124, spacing=0.01)**
```
Resampled: 333 waypoints (EE spacing=10.0 mm)
No collisions detected (333 waypoints)
Time profile: total=89.0s, max EE=50.0 mm/s, max rot=20.0 deg/s,
              max joint=0.30 rad/s, corners=120 seg
              (max angle=178.8°, slowdown=2.49x)
CSV: trajectory_dp_ee_s0010_eev50mms_av20dps_jv0p30_corner30d_x2p5.csv
reconfigs=14 (inter=7, intra=0), transit=7/7 OK, collisions=0
```

해석:
- 332개 segment 중 120개 segment 에 corner slowdown 적용.
- `max angle=178.8°` 는 거의 되돌아가는 수준의 sharp turn 이 존재한다는 뜻.
- 현재 설정은 흔들림 완화 쪽으로 보수적이며, 전체 시간이 89초로 늘어남.

튜닝 후보:
- 더 빠르게 하고 싶으면 `CORNER_ANGLE_THRESHOLD_DEG = 45.0`, `CORNER_MAX_SLOWDOWN = 2.0` 부터 테스트.
- 그래도 흔들리면 threshold 는 45로 두고 `CORNER_MAX_SLOWDOWN` 만 2.3 정도로 증가.

## 결정/변경

- **셸 환경 통일**: 모든 실행은 `ros-jazzy` 컨테이너 안. 호스트 venv 활성화 금지. 컨테이너 안에서도 root 셸로 실행 금지 (cache/USD 권한 오염).
- **InspectionCamera 부모 = SG8S Xform**: SG8S 내부 좌표계와 일관성. SG8S 가 wrist_3_link 자식이라 EE 체인은 자동 추종.
- **카메라 transform 은 SG8S 내부 기존 Camera 그대로 복사**: payload 가 정의한 광학 중심 / 회전을 단일 진실원으로 둠. payload 가 바뀌어도 자동 추종.
- **단일 ActionGraph**: 별도 그래프는 evaluator 가 매 틱마다 따로 돌아 부하 증가. 통합 후 OnPlaybackTick + ROS2Context 공유.
- **`CAMERA_PUBLISH_W/H` 분리**: 센서 스펙 ≠ 발행 해상도. 진실원/런타임 부하 분리.
