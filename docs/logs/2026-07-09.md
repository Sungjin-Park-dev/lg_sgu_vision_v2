## 핵심 발견 — 구 메시 앞부분은 실제 형상이 아니었다

`camera_body.obj` 를 실측해보니 **실제 CAD 몸체는 x = 0.141 에서 끝난다.** 그 앞
`0.141 → 0.346` 구간(205 mm)은 **정점 8개 / 면 18개짜리 손으로 만든 프록시 블록**이었다
(단면 120×100 mm, 좌표가 전부 `{0.266, 0.346} × {−0.0869, −0.0469, +0.0331} × {−0.046, +0.054}`
같은 딱 떨어지는 손입력 값). 조명 유닛의 조잡한 대역이다.

새 CAD 는 그 블록을 없애고, **이전엔 모델링조차 안 됐던 Ø38.4 mm 실제 렌즈 배럴**을
`0.141 → 0.21877` (78.8 mm)에 담고 있다. 광축 정중앙(y=0.000, z=0.000)이며 전체 면의 87.1 %.

→ 단순 절단이 아니라 **"프록시 블록 제거 + 배럴 추가"**. 신 메시는 구 메시의 부분집합이 아니다.
공유 몸체(x < 0.141)는 점-대-표면 mean 0.011 mm / p95 0.076 mm 로 동일.

MorphIt 이 이 블록에 씌워둔 구체가 29개 중 **22개**(반지름 최대 0.069 m)였다. 이게 물체 근처
자세를 대량으로 오검출 충돌 처리했고, transit 이 via-roll 로 우회하느라 106 s 를 썼다.

## 방향 정합

새 OBJ 는 광축이 +Z, 기존 규약은 flange +X. 회전을 **정점에 베이크**해 방향이 맞는 OBJ 를 새로 만든다
(USD 에도 베이크된 점을 넣고 `xformOp` 는 authoring 하지 않는다 — 기존 `camera.usdc` 규약).

```
old_x =  new_z        R = [[0,0,1],        det = +1
old_y = -new_x             [-1,0,0],       ≡ USD rotateXYZ(-90°, 0°, -90°)
old_z = -new_y             [0,-1,0]]
```
평행이동 없음(둘 다 마운트면이 −0.001). 부호 모호성(`old_z = ±new_y`)은 `det` 가 결정 —
`+new_y` 변형은 `det = −1` 인 거울상이라 불가. 적용 후 bbox 오차 **0.0 µm**.

## 한 일

### 1. `scripts/isaac/usd/bake_camera_mesh.py` (신규)

`ff/.../camera.obj` → 회전 bake → 데시메이션 → `camera.usdc` + `camera_body.obj` **동시 출력**.
두 산출물을 같은 in-memory 배열에서 뽑는다 — `camera_body.obj` 는 중간물이 아니라 URDF
visual+collision · MorphIt 입력 · `ik_inspector` 렌더 대상이라, 따로 만들면 반드시 어긋난다.
(검증: points 최대 차 5e-7 = `%.6f` 반올림, faces 완전 일치.)

- **데시메이션은 `open3d`.** `trimesh.simplify_quadric_decimation` 은 `fast_simplification`
  백엔드를 요구하는데 미설치 → `ModuleNotFoundError`. `open3d` 는 이미 `pyproject.toml` 의존성.
  `boundary_weight=1000` (메시가 non-watertight).
- 451,678 → **129,999 tri / 65,794 v** (구 파일 128,078 / 67,007 과 동급). bbox 오차 0.0 µm,
  표면적 100.00 %, 원본→데시 표면편차 mean 0.017 mm / p95 0.055 mm. OBJ 4.5 MB (구 4.5 MB).
- normals 는 **faceVarying, crease 30°** 로 재계산. 원본 OBJ 의 `f v//vn` 대응은 데시메이션으로
  깨지고, 단순 vertex-normal 평균이나 `open3d.compute_vertex_normals()` 는 x≈0.16 의 하드 림을 뭉갠다.
- `camera.usdc` 는 순수 `pxr` 로 저술 (Kit/SimulationApp 불필요). prim 은
  `/root/camera_body_geom/camera_body_mesh` 로 개명 — `CAMERA_ASM_ASM__Meshed__001` 은
  with-light 조립체 이름이라 오해를 부르고, 의존처가 0건임을 확인했다.
- **anti-`0c85601` assertion 내장**: 합성된 `ur20_with_camera.usd`/ghost 에서
  `camera_body` 하위 Mesh 가 정확히 1개, bbox 일치, mesh→flange identity,
  `camera_optical_frame` translate 가 여전히 `(0.346, 0, 0)` 인지 단언. `--verify-only` 로 재실행 가능.
  (`BBoxCache` 가 아니라 raw points 로 단언 — `ComputeRelativeBound` 는 ~6e-5 m 패딩을 준다.)

`ur20_with_camera.usd`(25.8 MB) 는 **손대지 않았다.** `camera_body` prim 이 리터럴
`./camera/camera.usdc` 를 reference 하므로 같은 파일명으로 덮어쓰면 참조가 그대로 살아난다.

### 2. `camera_link` collision sphere 재피팅

전체 재빌드가 아니라 **`--edit-config --refit-link camera_link`**:

```bash
STAGE=<scratch>/ur_asset_stage/ur_description/meshes   # package:// 가 호스트에서 resolve 안 됨
ln -s workcell/robot/ur20 "$STAGE/ur20"; cp camera_body.obj "$STAGE/camera/"
sed -e 's|urdf_path:.*|<abs>|' -e 's|asset_root_path:.*|<abs STAGE>|' \
    workcell/robot/ur20_with_camera.yml > refit_in.yml     # --asset-path 는 edit 경로에서 무시됨
.venv/bin/python -m curobo.examples.getting_started.build_robot_model \
  --edit-config refit_in.yml --refit-link camera_link --output ...gen.yml --export-xrdf --seed 42
```

- `builder_robot.py:200` 이 `robot_cfg` 를 자동 unwrap 하고, `:229`+`:637` 이 `cspace` 를 복사해
  그대로 되뱉는다 → 06-24 의 post-edit(`default_joint_position`/`max_acceleration`/
  `position_limit_clip` 복원) 이 **불필요**하다. 06-24 절차 갱신.
- `--recompute-collisions` 는 쓰지 않았다. 기존 `self_collision_ignore` 는 더 작아진 카메라에 대해
  superset 이라 안전하고, 재계산하면 ignore 가 빠져 영구 self-collision 이 생길 위험이 있다.
- **승격은 텍스트 splice**로 했다. 빌더 출력은 `collision_link_names` 를 알파벳순으로 재정렬해
  YAML 앵커(`&id001`/`*id001`)를 잃고 `urdf_path`/`asset_root_path` 에 절대경로를 박는다.
  `camera_link:` 블록만 갈아끼우면 나머지 151개 구체가 바이트 단위로 보존된다 (검증 완료).

## 주의 / 남은 것

- **`camera_optical_frame` 이 메시 끝(0.21877)보다 127 mm 앞 허공에 뜬다. 정상이다.**
  그 지점은 제거된 조명 박스의 앞면이었다. 렌즈는 움직이지 않았으므로 초점면(flange + 0.392 m)은
  불변이고, 그래서 `viewpoints.h5` / `ik` / `trajectory` / `OBJECT_PLACEMENTS` 를 하나도
  재생성하지 않았다. legacy URDF 의 `camera.stl` 도 이미 113 mm 떠 있다 — 전례가 있다. **고치지 말 것.**

- **미해결: 초점면 0.392 m 가 맞는가?** `config.py:20` 의 주석 처리된
  `CAMERA_WORKING_DISTANCE_MM = 250.0` 과 `0.141 + 0.250 = 0.391 ≈ 0.392` 는, 초점면이
  "**몸체면**(0.141) + 렌즈 WD 250 mm" 가정에서 나왔음을 시사한다. 렌즈 WD 250 mm 가 통상처럼
  **배럴 앞면**(이제 0.21877) 기준이라면 초점면은 `0.469` 여야 하고, 파이프라인은 78.8 mm 너무
  가까이 계획해온 셈이 된다. 배럴이 처음 모델링되면서 드러난 **선재 이슈**다.
  → 데이터시트/실측으로 확인 필요. 맞으면 `camera_optical_joint` xyz, USD optical 행렬,
  `CAMERA_WORKING_DISTANCE_MM` 을 함께 옮기고 h5/ik/trajectory 전부 재생성해야 한다.

- `OBJECT_PLACEMENTS`(config.py:99–128)는 구 충돌 형상에 맞춰 `optimize_placement.py` 로 튜닝된 값이다.
