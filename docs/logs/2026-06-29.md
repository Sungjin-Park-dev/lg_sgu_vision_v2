## 1. TRANSIT_MAX_ATTEMPTS 2→1
graph off 라 성공은 attempt 0 에 끝나고 실패만 재시도만큼 헛돈다 → 1 로 실패 대기 절반. coverage 회귀
없음 실측(양쪽 동일). sample 56→44s.

## 2. ★ BatchMotionPlanner + short-circuit
- **단일→배치**: 후보 leg 를 하나씩 순차 `plan_cspace` 하던 것을 cuRobo `BatchMotionPlanner.plan_cspace`
  로 여러 문제 GPU 동시 계산. `_plan_chunk`(≤bs, 빈 슬롯 home padding=자명성공) / `_plan_batch`(chunk 분할)
  신설. main 은 **Round 0 에서 전 경계 direct 일괄** → 실패 경계만 사다리.
- **결과 shape(실측, scratchpad/probe_batch_cspace.py)**: plan_cspace 는 seed best 1 로 collapse →
  `success (B,1)` · `interpolated_trajectory.position (B,1,T,dof)` · `interpolated_last_tstep (B,1)`,
  추출 `pos[i,0,:last[i,0]+1,:dof]`. `get_topk_seeds()`/`get_interpolated_plan()`는 배치서 못 씀.
- **★ short-circuit(결정적 레버)**: per-boundary 로 후보 전체(leg+bridge≤60)를 한꺼번에 배치하면
  **순차보다 느리다**(46s>44s) — 순차는 가까운 가교쌍 찾으면 즉시 멈추는데(easy 경계 ~3 plan) 배치는 60
  전부 계산하기 때문. 그래서 `_via_variant` 를 **joint-closest 순 chunk 로 풀되 가교쌍 나오면 즉시 중단**.
  선택 규칙(가장 가까운 가교쌍) 불변 = coverage 동일.

## 3. VIA_ROLL_MAX_REPS 6→4
한 경계 후보(leg 8 + bridge 24 = 32)가 **딱 1 chunk(`TRANSIT_BATCH_SIZE=32`)** 에 들게 맞춘 값 →
easy 경계 1 chunk 로 끝. **coverage 레버**라 까다로운 경계(curved 71→72·sample 37→39, known-hard)가
4 로도 roll 가교 유지됨을 확인. 더 어려운 물체 나오면 MAX_REPS(과 batch_size) 키워 안정성↑/속도↓ 절충.

## 시도→폐기
- `use_cuda_graph=True`: 비용이 **compute-bound**(plan_cspace 1 chunk≈2.2s)라 효과 없고 메모리만 더 써
  OOM(bs=32, Phase-1 IK 가 ~11GB 점유) → `False` 유지. (`empty_cache`/`gc` 도 그 11GB 는 활성 점유라 무효 → 제거.)
- 후보 전체 일괄 배치(short-circuit 없이): 순차보다 느림 → 폐기.
