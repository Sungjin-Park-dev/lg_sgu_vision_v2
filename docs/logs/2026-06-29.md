## 2. ★ BatchMotionPlanner + short-circuit
- **단일→배치**: 후보 leg 를 하나씩 순차 `plan_cspace` 하던 것을 cuRobo `BatchMotionPlanner.plan_cspace`

  신설. main 은 **Round 0 에서 전 경계 direct 일괄** → 실패 경계만 사다리.
- **결과 shape(실측, scratchpad/probe_batch_cspace.py)**: plan_cspace 는 seed best 1 로 collapse →
  `success (B,1)` · `interpolated_trajectory.position (B,1,T,dof)` · `interpolated_last_tstep (B,1)`,
  추출 `pos[i,0,:last[i,0]+1,:dof]`. `get_topk_seeds()`/`get_interpolated_plan()`는 배치서 못 씀.
- **★ short-circuit(결정적 레버)**: per-boundary 로 후보 전체(leg+bridge≤60)를 한꺼번에 배치하면
  **순차보다 느리다**(46s>44s) — 순차는 가까운 가교쌍 찾으면 즉시 멈추는데(easy 경계 ~3 plan) 배치는 60
  전부 계산하기 때문. 그래서 `_via_variant` 를 **joint-closest 순 chunk 로 풀되 가교쌍 나오면 즉시 중단**.
  선택 규칙(가장 가까운 가교쌍) 불변 = coverage 동일.

## 1. IK 사전계산 → 기각 (data)
"roll/tilt IK 를 on-demand 말고 사전계산" 검토. **전 viewpoint IK = 1.7s, via IK 빌드 = 0.09s** (이미

## 2. cross-boundary 배치 → 구현 후 기각 ★
pending 경계 후보를 round-robin 으로 한 GPU chunk 에 pool(`_plan_via_pooled`+legs_done 가드, 선택 규칙
보존 의도). 생성은 빨라짐(sample transit_pending 19→14s). **그러나 폐기**:
- **`plan_cspace` 가 batch-composition 의존** — 같은 leg/bridge 문제도 섞인 배치에서 다른(유효하나 다른)
  궤적·성공집합을 냄(seed 가 배치 위치 의존). pool 하면 "joint-최근접 완성쌍" 선택이 임의로 바뀜.
- 측정(전=순차, 후=pooled, 둘 다 결정적·coverage 동일): **실행 cycle** sample 216.9→172.3s(−20%),
  **curved 140.9→176.4s(+25%, +35s)** — 방향이 물체마다 제멋대로. scan 구간 byte-동일, 차이는 전부 transit.
- 결론: 생성 ~2-5s 절감이 **반복 실행 inspection cycle 을 임의로(때로 +25%) 악화** → 나쁜 거래. revert.
  **plan_cspace 를 재배치하는 어떤 transit 최적화도 같은 함정**(궤적 보존 불가). 순차 `_via_variant` 복원,
  baseline md5 일치 확인. (`_via_variant` docstring 에 NOTE 로 사유 기록.)

## 3. solver 재사용 → 채택 ★
`compute_fk`(Kinematics)·`batch_collision_check`(RobotCollisionChecker)가 **매 호출 cuRobo 객체를 새로
빌드**하고 있었음. run 내 robot_cfg·world_scene 불변 → `_get_kinematics`/`_get_collision_checker` id-캐시로
**run당 1회만** 빌드. 결과는 빌드 횟수와 무관 → **궤적 보존(sample·curved 둘 다 baseline md5 정확 일치)**.
