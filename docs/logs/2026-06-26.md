# 2026-06-26

`plan_trajectory.py` 궤적 생성 파이프라인 대수술. **카메라가 target object를 관통하는데도 저장되던 충돌검사
버그(근본 원인)를 잡고**, transit 강건화·실패 transit 안전 처리·reconfig 최소화·unreachable viewpoint
드롭까지 적용. curved_structure/100 기준 reconfig 11→3, 가짜 커버리지(carry-forward) 제거하고 정직한
커버리지를 출력하게 됨.

## 발단

`isaac_pipeline.py`로 생성한 궤적에서 **target object와 로봇 카메라가 충돌**하는데도 CSV가 저장됨.

## 1. (근본 원인) 충돌검사 부호 반전 — `batch_collision_check`

- cuRobo 충돌 cost는 **음수가 안 됨**: `0=안전`, `>0=충돌`. cuRobo 본체 `RobotSceneCollision.validate()`도
  "충돌 없음 ⇔ summed cost == 0.0"으로 판정(`collision_robot_scene.py:288`).
- 그런데 코드가 `d_scene < 0`, `d_self < 0`을 검사 → **절대 참이 안 됨** → 월드/자가 충돌이 **상시 무력화**.
  "No collisions detected"는 의미 없었고, 카메라가 물체를 관통해도 그대로 저장됨.
- **수정**: `< 0` → `> 0`(eps 포함). 추가로 `RobotCollisionCheckerCfg.load_from_config`의
  `collision_activation_distance` 기본값이 **0.2 m**라 `>0`만 고치면 작업거리 46mm 카메라가 늘 그 안에
  들어 *모든* waypoint를 충돌 판정 → 최종검사 activation을 `config.COLLISION_MARGIN`(0)으로 명시해
  `cost>0 ⇔ 실제 침투`가 되게 함.

## 2. (#1) Reconfig transit 강건화 — `plan_reconfig_transits`

- `warmup(enable_graph=False)` → **`enable_graph=True`**로 PRM roadmap까지 워밍업. reconfig transit은 자세를
  크게 뒤집는 hard 케이스라 trajopt 직선 시드만으론 자주 실패 → graph seeding이 핵심.
- 재시도 `max_attempts` 10 → **`TRANSIT_MAX_ATTEMPTS=15`**. 효과: 실패 3 → 1.

## 3. (#2) 실패 transit 안전 처리 — `interpolate_and_resample`

- 기존엔 transit이 실패한 reconfig를 **joint-space 직선 보간**으로 메움 → 큰 점프(예: 112°)를 직선으로 쓸어
  물체를 관통. (1번 버그로 그동안 안 잡혔던 것.)
- **변경**: 직선 보간 금지. 경로를 '안전 연속 run'들로 분할(`_build_runs`)하고 **가장 긴 run 채택**.
  고립 아웃라이어는 `TRANSIT_FAIL_SKIP_MAX`개까지 건너뛰어 재연결, 이을 수 없는 경계는 run을 끊음.

## 4. (b)+wrist_3 — reconfig 대폭 감소

진단 추가: 각 intra-cluster reconfig마다 어떤 joint가 튀는지 + "연속 해가 IK pool에 있었는지" 출력.
이걸로 49→50의 174° flip이 **wrist_3 때문**임을 발견.

- **wrist_3를 DP '이전'에 잠금** (기존: DP 후). wrist_3(광축 roll)는 어차피 0 고정인데, IK는 free wrist_3로
  풀려 후보마다 제각각이고 DP reconfig 비용은 6-DoF L∞라, 5-DoF로는 연속인 해가 '버려질' wrist_3 차이로
  reconfig 오판됨. 미리 0으로 잠그니 DP가 실제 최종 자세로 연속 해를 고름.
- **(b) DP 후보 풍부화** (`ENRICH_DP_CANDIDATES`): DBSCAN medoid(클러스터당 1개) 대신 fine-tol(`0.08 rad`)
  dedup으로 분기 보존. DP 갱신 벡터화.
- 효과: **reconfig 11→3, mean jump 18.4°→7.0°, 174° flip 소멸.**

## 6. Unreachable(empty) viewpoint 드롭 — `DROP_UNREACHABLE_VIEWPOINTS`

- IK 해가 0개인 viewpoint를 carry-forward(이웃 자세 복제)로 메우던 것을 **경로에서 제거**로 변경. 복제는 그
  표면점을 실제 검사하지 않는 **가짜 커버리지**라, 도달 가능한 viewpoint만 남겨 정직하게 검사.
- `orig_idx`로 드롭 후에도 모든 로그(reconfig 진단/transit/run-split)를 **원본 viewpoint 번호**로 표기.
- 최종 요약에 `coverage=NN/100 (unreachable dropped, transit-split dropped)` 출력.

## 7. (검증) "고정 visit order(GTSP)가 reconfig 유발" 가설 — **기각**

가설: GTSP는 EE 이동거리만 최적화하고 IK 분기는 모르니, 공간 인접 viewpoint가 정반대 arm 자세를
요구하면 스캔 한복판에서 flip이 강제된다 → 순서를 분기 인식으로 재정렬하면 reconfig가 줄 것이다.

**검증 방법**(`scratchpad/order_analysis_multi.py`): 같은 reps·같은 `dp_optimal_path`로, **순서만** 3가지로
바꿔 reconfig를 사과 대 사과로 비교. (1) GTSP 고정, (2) 연속성 greedy 재정렬, (3) IK 분기 family
(5-DoF L∞ threshold 내 해 공유 그래프의 연결성분)로 그룹핑한 순서. 전 케이스 **실제 WD 46mm로 통일**.

| 케이스 | 도달 | GTSP-DP | greedy-DP | family-group-DP | 이론하한(family−1) | family 쪼개짐 |
|---|---|---|---|---|---|---|
| curved/100 | 94/100 | **5** | 5 | 5 | 4 | 0 |
| curved/80 | 47/80 | **3** | 4 | 3 | 3 | 0 |
| sample/74 | 55/74 | **3** | 4 | 3 | 3 | 0 |

- **family-group == GTSP (전부 동일)**: 분기 인식 재정렬로 reconfig가 한 개도 안 준다. `쪼개짐=0`이 이유 —
  **GTSP가 이미 각 IK 분기 family를 연속 블록으로 배치**하고 있어 inter-family 경계가 최소(=family−1)다.
- **greedy ≥ GTSP (전부)**: 자유 재정렬은 무익, 두 케이스(80·74)에선 오히려 reconfig 악화(3→4)+EE 경로 증가.
- **80·74는 GTSP == 이론하한**: 이미 물리적 최소라 어떤 순서로도 더 못 줄임. curved/100만 하한+1인데, 이는
  한 family **내부** reconfig(연결돼 있으나 threshold 내 단순경로 없음)라 순서가 아니라 그 분기의 IK 구조 문제.
- (주의) 250mm로 잘못 돌렸던 1차 시도에선 80에서 greedy 7→5로 보였지만 **WD 불일치 착시**였고, 46mm로 맞추니 역전.
- cylinder/21은 46mm·해당 배치에서 IK 0%(전부 도달 불가) → 순서와 무관, "placement가 레버"를 보강.

**결론**: 순서 고정(GTSP)은 reconfig의 원인이 아니다. reconfig는 곡면+46mm에서 IK 분기가 갈라지는 기하학적
사실이고 GTSP는 이미 그 분기를 최적 배치 중이다. 분기 인식 재정렬/멀티-패스는 **이득 0, EE 경로만 악화** 위험이라
보류. 진짜 레버는 일관되게 **reachability(물체 배치)**.
