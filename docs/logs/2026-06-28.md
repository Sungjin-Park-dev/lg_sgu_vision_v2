## 1. via-roll/via-tilt — transit-fail 해결

- **문제**: 좋은 배치에서 reachable 이 커지면 reconfig transit 이 일부 실패 → `_build_runs` 가 최장 run 만
  남기고 나머지 reachable 을 버림(curved 좋은배치 visited 66/99).
- **원인**: 카메라 광축 ≈ wrist_3 축 → wrist_3 회전 = 광축 roll = 검사 무손실 **자유 redundant DOF**.
  그런데 pipeline 이 `r[:, -1]=HOME값` 으로 wrist_3 을 상수 lock 해 이 자유도를 버려, 인접 viewpoint 로
  가려면 다른 관절이 272° 휘두르는 hard reconfig 발생.
- **해법**(`plan_reconfig_transits` 사다리): `direct → via-roll → via-tilt → via-home`.
  via-roll = 경계 양 끝 scan 자세를 광축 둘레로 roll 한 중간자세를 경유(`sel[i]→A'(roll)→A'→B'→B'→sel[i+1]`).
  scan config 보존이라 DP·스캔·`_build_runs` 무수정(ripple 없음). 측정: curved 좋은배치 **66→99/99**,
  collisions=0. (상세: 메모리 `plan-trajectory-via-roll-transit`)

## 4. ★ 생성 시간 최적화 — graph(PRM) 제거

### 측정 (생성 wall-clock, IK 재계산 포함)
| 물체 | 전 | 후(graph off, max_attempts=2) | coverage | collisions |
|---|---|---|---|---|
| curved | ~2분+ | **27.5s** | 99/100 (동일) | 0 |
| sample | 4:27 | **56s** | 69/74 (동일) | 0 |

- **분해**: Phase1 IK ~1.3s(무시), 거의 전부 Phase4 transit. transit 비용은 trajopt 가 아니라 **PRM
  graph search** 가 지배(실패하는 hard reconfig 가 graph 로 우회로 탐색하며 느림).
- **핵심 발견**: graph 는 빡센 reconfig(측면 flip)의 "직선 시드 물체 관통" 문제를 우회로로 풀려고 켰던
  건데, **via-roll/via-tilt 가 같은 문제를 rolled 중간자세 경유로 이미 해결** → graph 가 **redundant**.
- **변경**: `TRANSIT_ENABLE_GRAPH_ATTEMPT=99`(>max_attempts → graph 미사용) +
  `warmup(enable_graph=False)` + `TRANSIT_MAX_ATTEMPTS 8→2`(graph off 면 성공은 attempt 0 에 끝, 실패만
  헛돌아 낮게). → **5~9배 단축, coverage·collisions 동일**.
- **부정 결과(배치 시도, 폐기)**: cuRobo `BatchMotionPlanner.plan_cspace` 로 후보 leg 를 GPU 배치 병렬화
  시도 → **실패**. 병목이 trajopt 가 아니라 **graph search** 라 GPU 배치 병렬화가 안 먹히고 오히려 6.5분.
  (배치 결과 shape 함정: `(B,goalset=1,T,dof)`/`success(B,1)`/`last_tstep(B,1)` → `[i,0]` 추출 필요.
  이걸 빠뜨려 한때 0/9 오판 + 40분 cascade 했음.)

## 5. ★ 다음 세션 — 시간 더 줄이기 (아이디어)

현재 병목: **via-roll/via-tilt 의 경계내 후보 탐색이 순차**(경계당 ~20 pair × 최대 3-leg 를 trajopt 로
한 번에 하나씩). curved 27.5s 는 목표(20~30s) 달성, **sample 56s 는 via-tilt 가 필요한 빡센 경계 1개**
(37→39)가 지배(roll 전부 실패→tilt 탐색).

1. **★ trajopt 배치 재시도 (top lead)**: 배치 실패는 graph 때문이었다. **이제 graph 가 off 라 trajopt-only
   이고, trajopt 는 GPU 배치 병렬화가 잘 된다.** via-roll bridge 후보쌍(≤49) + roll-leg 를 한 배치
   plan_cspace 로 묶으면 순차 탐색이 붕괴할 수 있음. 추출 함정(goalset 차원 `[i,0]`)만 주의. batch_size 는
   15GB GPU 에서 32 가 한계권(64 는 OOM). **sample 56s 를 sub-30s 로 끌 가장 유망한 레버.**
2. `TRANSIT_MAX_ATTEMPTS 2→1`: graph off 라 성공은 attempt 0 에 끝나므로 1 로도 coverage 유지될 가능성.
   (3→2 는 sample 1:07→0:56. 1 은 미검증 — coverage 확인 필요)
3. 후보 수 축소: `VIA_ROLL_PAIR_CAP`(현재 미사용/20), `ROLL_VARIANT_DEG`(7개), `VIA_ROLL_IK_SEEDS`(40) 줄이면
   탐색·변형IK 가 빨라지나 가교쌍 누락 위험 → coverage 측정 필수.
4. via-tilt 가 필요한 경계에서 **roll 전탐색이 헛수고**(다 실패). roll 이 가망 없음을 빨리 감지해 tilt 로
   조기 전환하는 휴리스틱(예: min-pair joint 거리/예비 1쌍 테스트로 판단).
5. GPU 메모리: Phase1 IK solver 를 transit 전에 해제하면 배치 batch_size 를 키울 여지(현재 ~12GB 점유).
6. (확정 무효 레버) 클러스터 순서/재정렬, 배치는 coverage/속도 레버 아님 — 건드리지 말 것.
